## 1. 핵심 페르소나: Context-Complete Reporter

> *"3개월 후의 나, 그리고 AI가 이 한 줄만 봐도 전체 맥락을 이해할 수 있도록 쓴다"*

```
지금의 나  → 모든 맥락을 알고 있음
3개월 후의 나 → 이 업무 기억 못함
AI          → 이 문서가 전부임
```

세 경우 모두 **같은 수준의 이해**가 가능해야 함 → 지금만 아는 맥락을 기대하면 안 됨

## 2. 기본 구조 템플릿 (CRM 입력용)

```
[업무유형]  ← 기술지원 / 고객상담 / 기타 / 연구개발 / 영업

▶ [고객사 풀네임] - [제품명] [구체적 이슈명]
∙ [고객사_부서명 전체] [담당자 이름] [직책]

* 문제/배경: [무엇이 문제였는지 한 문장으로]
- 원인: [파악된 원인 또는 분석 결과]
- 조치: [내가 취한 구체적 행동]
- 결과: [완료 / 진행중 / 보류 / 고객 회신 대기]
```

## 3. 작성 예시

### ❌ 나쁜 예 (기존 스타일)

```
[기술지원]
▶ 삼성 MX - ramdump 문의
∙ 임영재 Principal

* zram stack 정보 미표시 확인
- 최신 SW에서도 동일 문제 발생
- task.process 성공적으로 보임
```

> 문제: "어떤 제품?" "어떤 SW?" "결과는?" → AI가 이 청크만 봤을 때 맥락 없음

### ✅ 좋은 예 (권장 스타일)

```
[기술지원]
▶ 삼성전자 시스템LSI사업부 - TRACE32 ramdump kernel 6.18 module 주소 오류 문의
∙ 삼성전자_시스템LSI사업부 AP SW개발팀 최정호 / MX사업부 System개발2그룹 임영재 Principal

* 문제: TRACE32 ramdump에서 kernel 6.18 환경의 zram module .text section 주소 미표시 현상 발생
- 원인: Symbol.Autoload가 .plt section을 우선 참조하도록 설계된 정상 동작으로 추정
- 조치: y.list.section <module address> 명령으로 .text section 실제 주소 데이터 확인 요청
- 결과: 고객사 측 확인 후 재연락 예정 (진행중)
```

> 효과: "ramdump", "kernel 6.18", "삼성전자", "zram", "TRACE32" 어떤 단어 검색에도 전체 맥락 제공

## 4. 5가지 작성 규칙 체크리스트

작성 전 아래를 확인하세요:

- [ ] **① 고유명사 풀네임**: 고객사 부서명까지 전체 표기 (삼성전자_시스템LSI사업부 AP SW개발팀)
- [ ] **② 제품명 명시**: 이슈가 발생한 제품/기술 명칭 (TRACE32, ramdump, kernel 6.18)
- [ ] **③ 인과관계**: 문제 → 원인 → 조치 → 결과 흐름으로 작성
- [ ] **④ 상태 명시**: 마지막 줄에 반드시 `결과: 완료 / 진행중 / 보류 / 고객 회신 대기` 중 하나 표기
- [ ] **⑤ 업무명 핵심어**: ▶ 줄에 고객사 + 제품명 + 이슈 핵심어 포함

## 7. RAG 최적화 원리 (NotebookLM / 지식 검색용)

### RAG란?

**RAG (Retrieval-Augmented Generation)**:
```
① 질문을 받음
② 저장된 문서에서 관련 청크(chunk)를 벡터 검색으로 꺼냄
③ 꺼낸 청크를 근거로 답변 생성
```

→ **"어떤 청크를 꺼내오느냐"** 가 답변 품질을 결정  
→ NotebookLM은 MD 파일을 **H2(##) 단위**로 청킹

### 청크 설계 원칙 5가지

**원리 1 - 고유명사 풀네임**: AI는 "삼성"과 "삼성전자"를 다른 개체로 인식
```
X: 삼성 MX, T32, LSI
O: 삼성전자_MX사업부, TRACE32, 삼성전자_시스템LSI사업부
```

**원리 2 - 업무명(▶)에 핵심 검색어 포함**: 청크 제목에 검색어가 있어야 검색됨
```
X: ▶ 삼성전자 MX사업부 - ramdump 문의
O: ▶ 삼성전자 MX사업부 - TRACE32 ramdump kernel 6.18 module 주소 오류 문의
```

**원리 3 - 인과관계 명시**: 문제→원인→조치→결과 흐름 필수
```
X: - task.process 정보는 성공적으로 보임
O: - 조치: y.list.section 명령으로 .text section 주소 확인 요청
   - 결과: 고객사 확인 후 재연락 예정 (진행중)
```

**원리 4 - 날짜·담당자 청크 내 반복**: 청크 분리 후 파일명(날짜)은 사라짐  
→ `save_all_reports.py`가 자동으로 처리

**원리 5 - 청크 크기 = 업무 1건**: 업무 1건(▶ 하나) = 청크 1개가 최적

### 전체 파이프라인

```
CRM 입력 (Context-Complete 페르소나로 작성)
  → 풀네임 + 제품명 + 문제→원인→조치→결과
        │ CRM 자동 발송 (crm-noreply)
        ▼
Outlook "나의 일일보고" 폴더
        │ python save_all_reports.py (매일 1회)
        ▼
reports/YYYY_H?.md
  - 업무 1건 = ## 섹션 = RAG 청크 1개
  - 날짜/담당자/고객사/상세 자기완결
        │ 업로드
        ▼
NotebookLM / RAG 시스템
  → 정확한 청크 반환 → 고품질 답변
```
